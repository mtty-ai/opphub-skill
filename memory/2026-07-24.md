## 上午 — 修复 radar 显示乱码

### 起因
睿驰嘉禾录入完成后,刷新 knowledge 页,能力雷达 (radar) 显示:
- "蓝色光标"
- "宝马等一线上市公司"  
- "在用户运营"
- "整合营销"

全是公司名/团队背景信息, 不是能力词。

### 排查
1. **最初错改前端**: 我把 `app/knowledge/page.tsx` 的 `extractKeywords` 里 `pf.businessDescription.split(...)` 删了,加长度限制。
   - 舟哥骂: "**你丫用前端来处理关键词？**"
2. **正确诊断**: 应该是后端 skill 输出的 `(证据词: ...)` 有问题, 而不是改前端去过滤
3. **根因**:
   - `bin/opphub-knowledge-card.js` 的 `hasEvidence()` 用模板写死的 `purpose` 字符串 (e.g. "KOL 投放 / 媒介代理 / KOL 资源对接") 当证据词源
   - 匹配上了 `hit.evidence` 但 **没传给 `buildCard`** ← 漏传参数!返回 `(证据: rawText 包含 "${dim}")` 兜底
   - 前端 `extractKeywords` 的 regex `/证据词:\s*(.+?)\)/` 解析 `(证据: ...)` 格式 → 解析失败
   - 前端 fallback 用 `pf.businessDescription.split(...)` 拆业务描述 → 拆出"蓝色光标"等噪音

### 修复
1. **`extractDimDesc`**: 从 rawText 抽每个维度对应描述段落 (4 种格式: `dim(...)`, `dim: ...`, `### dim\n...`, 模糊匹配)
2. **`extractEvidenceFromDesc`**: 从描述文本挖关键词当 evidence
   - 按 `[,，、。；：:等和与的]+` 标点拆
   - 滤前导停用字 `从|到|为|全|由|覆盖|涵盖|含|以及|在|之|对|其|按`
   - 限 2-6 字 (中文名词性片段)
   - 滤数字开头 / 纯字母 3+ / `的|了|吗`
   - 拆不出有效词时降级到 dim 自身
3. **`buildCard`**: 去 `evidenceSource` 参数,统一输出 `(证据词: kw1, kw2)` 格式
4. **`hasEvidence`**: 顺序反转 → 先 purpose 关键词再 dim 兜底; 移到后面跑
5. **`GENERIC_TERMS` 清理**: 删 `达人营销/短视频/内容制作/平台代运营/电商转化/虚拟人/IP` (这些是具体维度名,不是真通用词)
6. `extractParsedFields.city`: 限定到「地址字段 || 工商信息节」,防止同业联盟里 `无忧传媒(北京...` 污染

### 教训 (从这次实施)
1. **subagent 偏离 plan 时 inline 修** — 比派 fix subagent 快
2. **e2e 一定要用生产 secret 测** — 本地用 `dev-secret-change-me`, 生产是 `jM5nX3kP8rQ2vT9wL6yB1eF4gH7aZcD0sU`, 401 invalid_token 第一时间暴露
3. **生产容器 mount 路径 ≠ git working tree** — `/opt/1panel/www/opphub-server/` 是另一个项目, 实际是 `/opt/1panel/www/opphub-web/` mount 到 `/app`
4. **submodule vs 独立 repo** — opphub-web 是真 submodule (主仓 .git 跟踪), opphub-ws 是独立 repo (主仓不跟踪, 自带 .git)
5. **DB schema ≠ Prisma schema** — v0.7.40 ws-server 升级了 prisma schema 加 ackedAt 字段, 但生产 DB 没跑 migration
